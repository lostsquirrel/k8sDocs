Ingress 控制器

要使一个 Ingress 工作的前提是集群中必须要有一个 Ingress 控制器在运行。

与其它类型的控制器作为 kube-controller-manager 的一部分运行不同， Ingress 不会自动在集群中 运行。 使用本文选择最适合你的集群的 Ingress 控制器实现。

k8s 项目目前支持和维护 GCE 和 nginx 控制器

其它的控制器

  • AKS Application Gateway Ingress Controller 是使用 Azure Application Gateway 为 AKS 集群 提供入口的 Ingress 控制器
  • Ambassador API 网关 是一个基于 Envoy 的 Ingress 控制器 有 社区 支持和 来自 Datawire 的 商业 支持
  • AppsCode Inc. 提供了最广泛使用的基于 HAProxy 的 Ingress 控制器 Voyager
  • AWS ALB Ingress Controller 使用 AWS Application Load Balancer 实现 Ingress
  • Contour 由 VMware 提供和支持的基于 Envoy 的 Ingress 控制器
  • Citrix 为它的 baremetal 和 cloud 上部署的硬件(MPX),虚拟化(VPX)和 free containerized (CPX) ADC 提供的 Ingress 控制器
  • F5 Networks 为 F5 BIG-IP Container Ingress Services for Kubernetes 提供 支持和维护
  • Gloo 是基于 Envoy 的开源 Ingress 控制器 提供了来自 solo.io 企业支持的 API 网络功能
  • HAProxy Ingress 用于 HAProxy 调度可定制的社区驱动的 Ingress 控制器
  • HAProxy Technologies 为 HAProxy Ingress Controller for Kubernetes 提供支持和维护。 详见 官方文档
  • 基于 Istio 的 Control Ingress Traffic
  • Kong 为 Kong Ingress Controller for Kubernetes 提供 社区 或 商业的 支持和维护 NGINX Ingress Controller for Kubernetes.
  • NGINX, Inc. 为 NGINX Ingress Controller for Kubernetes 提供支持和维护
  • Skipper 为 Service 提供 HTTP 路由 和反向代理。还包含了类似 k8s Ingress 的使用场景， 设计上可以作为自定义代理的库
  • Traefik 是个功能齐全的 Ingress 控制器 (Let’s Encrypt, secrets, http2, websocket) 还有来自 Containous 的商业支持

Using multiple Ingress controllers

用户可以在集群中部署 任意数量的 Ingress 控制器 如果集群中有多个 Ingress 控制器，在创建 Ingress 时需要为每个 Ingress 标注恰当的 ingress.class 来指示使用哪个 Ingress 控制器。

如果在 Ingress 中没有定义控制器类型，云提供商可能使用一个默认的 Ingress 控制器。

理想情况下，所有的 Ingress 控制器都应当满足这个规范，但是很多 Ingress 控制器动作方式都有点不同。

说明： 一定要仔细阅读你选择的 Ingress 控制器的文档，确保理解了相关注意事项。
